NOTES SUMMARIZER - SOFTWARE REQUIREMENTS SPECIFICATION (SRS)

1. FUNCTIONAL REQUIREMENTS (FR)

FR1: Multi-format Document Ingestion
The system shall support the upload of academic notes in PDF, JPG, and PNG formats, handling files up to 450MB and PDFs up to 1950 pages.

FR2: Hybrid OCR Extraction
The system shall extract text from uploaded documents using a multi-tiered pipeline: Azure Document Intelligence (primary) and PaddleOCR (local fallback) to ensure reliable text recovery from both printed and handwritten notes.

FR3: AI-Powered Text Sanitization
The system shall automatically clean extracted OCR text by removing non-educational noise (page numbers, signatures, university headers) using a combination of regex pre-filtering and LLM-based sanitization.

FR4: Adaptive Multi-Level Summarization
The system shall provide three user-selectable summarization depths—Concise (brief), Balanced (medium), and Detailed—utilizing outline caching to optimize processing speed across levels.

FR5: Algorithmic Quiz Generation
The system shall generate interactive multiple-choice questions (MCQs) with three difficulty tiers (Beginner, Intermediate, Advanced), including correct answers and educational explanations.

FR6: Hierarchical Visualization
The system shall generate syntactically valid Mermaid.js code to render mindmaps and concept relationship graphs based on the structural hierarchy of the notes.

FR7: Contextual Study Modes
The system shall provide three distinct cognitive workflows—Quick Revision, Deep Study, and Exam Preparation—each offering a unique subset of AI-generated study tools tailored to the user's specific learning goal.

FR8: Grounded AI Tutor
The system shall provide a conversational interface (AI Tutor) that allows users to ask questions grounded strictly within the context of the uploaded document.

FR9: Resource Mapping
The system shall extract core concepts and generate optimized search queries to recommend relevant external educational resources (YouTube videos) to supplement the notes.

FR10: Structured Fact Extraction
The system shall identify and extract formulas, equations, and key factual definitions into a structured "Cheat Sheet" or "Last-Minute Revision" format.


2. NON-FUNCTIONAL REQUIREMENTS (NFR)

NFR1: High Availability (Reliability)
The system shall ensure operational continuity by implementing automated failover mechanisms (fallbacks) for all critical external API dependencies (OCR, Cleaning, and Summarization).

NFR2: Latency Optimization (Performance)
The system shall minimize user wait times for complex generation tasks by employing a parallel "racing" architecture, where multiple LLMs are queried simultaneously and the fastest valid response is accepted.

NFR3: Data Sovereignty (Privacy)
The system shall maintain a stateless backend architecture, ensuring that no user-uploaded files or extracted intellectual property are persisted in a database or stored beyond the immediate request lifecycle.

NFR4: Concurrency and Scalability
The backend shall utilize asynchronous I/O and non-blocking process execution (FastAPI/asyncio) to support concurrent file processing and feature generation without performance degradation.

NFR5: Graceful Degradation
The system shall be designed to provide partial functionality (e.g., raw text extraction) even if high-tier AI summarization services are temporarily unavailable or rate-limited.

NFR6: Responsiveness and UX
The frontend shall provide real-time stage-based navigation and visual feedback (progress toasts and loading states) to keep the user informed during long-running background processes.

NFR7: Security (Injection Mitigation)
The system shall sanitize all user-provided document text before inclusion in LLM prompts to mitigate the risk of prompt injection attacks or unauthorized system command execution.


3. SOFTWARE DEVELOPMENT LIFECYCLE (AGILE PROCESS)

3.1 Requirement Analysis & Planning (Sprint 0)
The project utilized a student-centric backlog, translating user needs into granular user stories. Agile planning prioritized the core document ingestion and "Quick Revision" mode first, ensuring a Minimum Viable Product (MVP) could be validated early while allowing for flexible adjustments to "Exam" and "Deep" modes based on initial feedback.

3.2 System Design (Architecture)
A decoupled, modular architecture was designed to support Agile's iterative nature. By using FastAPI for the backend and React for the frontend, the team could independently evolve the OCR pipeline and AI services. This modularity ensured that new features, like Mermaid.js visualizations, could be integrated into the existing framework without requiring a full system redesign.

3.3 Implementation (Iterative Sprints)
Development was executed in time-boxed sprints focusing on functional increments. Early sprints established the "Walking Skeleton" (file upload and raw OCR), while subsequent iterations incrementally added complexity, such as LLM-based sanitization and algorithmic quiz generation. This allowed for continuous integration and the ability to pivot technical strategies (e.g., switching LLM providers) mid-development.

3.4 Testing & Review (Continuous Validation)
Testing was integrated into the definition of "Done" for every sprint. Unit tests for AI pipelines and integration tests for the hybrid OCR system were run continuously. Sprint reviews focused on both technical accuracy (OCR confidence) and user experience (frontend responsiveness), ensuring that each increment moved the product closer to the students' educational goals.
